Camelot OS Render Update Manual and Implementation Prompt

Prepared for: David Goldoff and Sam Lodge

Date: June 16, 2026

Purpose: Turn Camelot OS into a stable, revenue-driving property intelligence system with HubSpot as the daily sales workspace, Supabase as the fact and automation layer, and Render as a fast front-end viewer.

Bottom line: Camelot OS should stop trying to do everything inside the browser. The browser should trigger jobs and display results. Supabase should own facts, reports, archives, jobs, and HubSpot sync. HubSpot should own leads, contacts, companies, deals, tasks, and follow-up.

1. Plain-English Manual

The Three-System Rule

SystemJobWhat It Should Not Do
HubSpot Daily workspace for sales, interns, cold callers, contacts, companies, deals, calls, notes, tasks, meetings, and pipeline. It should not be the place where raw public-record facts are guessed or manually retyped without source control.
Camelot OS / Supabase Property intelligence engine: verified building facts, fact packets, scoring, report archives, scheduled jobs, and HubSpot sync. It should not become a messy CRM replacement.
Render Frontend Fast user interface: search, view records, trigger jobs, preview reports, download documents. It should not generate heavy PDFs, scrape sources, run scoring, or freeze the user browser.

Why Camelot OS Has Been Breaking

The system is currently overloaded. Jackie, Daily Hunt, proposals, PDF rendering, HubSpot syncing, images, and facts are all too tightly mixed together. That creates the same failures again and again: wrong addresses, stale manager names, wrong unit counts, missing images, frozen pages, blank PDFs, and buttons that look real but call nothing reliable.

The fix is not another button. The fix is separating the work into smaller tools with clear responsibilities.

Fact Packet Rule

Every property must have a versioned fact packet. Reports should not re-research or re-guess facts at render time. They should read from the latest approved fact packet.

Every key fact should have:

  • value
  • source
  • retrieved date
  • confidence level
  • human override note, if applicable

This is how Camelot fixes the recurring “why is the address wrong?” and “why is the unit count wrong?” problem. Fix it once in the fact packet; every report then uses the corrected version.

PDF and Image Rule

Do not generate heavy PDFs in the browser. Generate reports server-side, save them to Supabase Storage or Drive, and show the finished PDF link in the browser.

Images should be cached and resized before reports use them. Do not embed large base64 images inside report HTML. Use optimized URLs and thumbnails.

HubSpot Rule

HubSpot should receive every qualified building, contact, deal, task, note, report link, and follow-up. Interns and cold callers should work in HubSpot, not inside Camelot OS.

Minimum HubSpot sync for each qualified lead:

  • Company: building record
  • Contacts: decision-makers and contact paths
  • Deal: property management opportunity
  • Task: next action
  • Note: Camelot OS intelligence summary
  • Report link: latest PDF or report archive

Revenue Priority

The first revenue win is not a prettier report. The first win is a reliable daily pipeline: Daily Hunt finds leads, HubSpot assigns calls, Camelot OS provides intelligence, and humans book meetings.

The second revenue win is TARA: every billable template task gets queued, reviewed, invoiced, archived, and potentially included in staff incentive compensation after the fee is collected.

2. 30-Day Implementation Plan

Week 1: Stop the Bleeding

  • Hide or disable buttons that do not call real backend functions.
  • Stop browser-side PDF generation for large reports.
  • Create tables for fact packets, reports, job runs, media assets, HubSpot sync logs, and TARA billable tasks.
  • Add a dashboard banner showing whether Daily Hunt last ran successfully.

Week 2: Build the Fact Packet Layer

  • Create the Fact Authority function.
  • Store source-proven facts by property.
  • Add human override fields for corrected address, unit count, manager name, building type, and images.
  • Make every report read from the same packet.

Week 3: Fix Reports and Images

  • Move report generation server-side.
  • Cache and optimize property images.
  • Generate First Email, Meeting Handout, Full Appendix, and Proposal from the same facts.
  • Archive every generated report with property, date, user, and version.

Week 4: HubSpot and Revenue Workflow

  • Push qualified buildings to HubSpot as companies.
  • Create/update contacts and deals.
  • Create follow-up tasks.
  • Attach report links to HubSpot notes.
  • Queue TARA billable tasks and invoice drafts.

3. Implementation Prompt for Codex / Replit

Use the following prompt to update Camelot OS or build the Replit prototype:

You are building the next stable version of Camelot OS for Camelot Property Management.

Goal:
Separate Camelot OS into clean modules so the platform stops freezing, stops generating wrong property facts, and starts producing revenue through HubSpot lead workflow and TARA billable task automation.

Core architecture:
1. HubSpot is the daily workspace and system of record for companies, contacts, deals, tasks, calls, notes, meetings, and pipeline.
2. Supabase is the property intelligence layer: fact packets, reports, media cache, job runs, HubSpot sync logs, and TARA billable tasks.
3. Render is the static frontend: it displays data, triggers backend jobs, previews reports, and downloads PDFs. It does not perform heavy scraping, PDF rendering, scoring, or HubSpot sync in the browser.

Build these modules:

A. Fact Authority
- Create versioned fact packets for each building.
- Every fact must include value, source URL, retrieved date, confidence, and override notes.
- Reports must read from fact packets and never re-derive facts at render time.

B. Report Renderer
- Generate First Email Intro, Meeting Handout, Full Appendix, and Proposal from the same fact packet.
- Use server-side rendering for PDFs.
- Store generated PDFs and HTML outputs in an archive.
- Add buttons: Preview, Download PDF, Email PDF, Push to HubSpot, Print.

C. Media Desk
- Upload, cache, resize, and reuse building photos.
- Convert large images to optimized WebP/JPEG.
- Do not embed large base64 images in reports.
- Every report should support multiple property and neighborhood images.

D. Daily Hunt
- Run as a backend scheduled job.
- Find, dedupe, score, and store building leads.
- Log each run in job_runs.
- Push qualified leads to HubSpot.

E. HubSpot Sync Desk
- Create/update building as Company.
- Create/update decision-makers as Contacts.
- Create/update property management opportunity as Deal.
- Create next-step Tasks.
- Add intelligence Notes and report links.
- Store HubSpot IDs back in Camelot OS.

F. TARA: Template Automation & Revenue Agent
- Manage templates, fee codes, billable task queue, invoice drafts, and generated document archive.
- Classify requester and payer role: association, board member, unit owner, shareholder, resident, renter, buyer, seller, vendor, landlord, broker, owner, prospect.
- Never auto-bill. Queue for manager approval.
- After approval, create invoice draft and archive the document.
- Support staff incentive tracking up to 30% of approved and collected fees.

Hard rules:
- Do not generate heavy PDFs in the browser.
- Do not let the frontend call dead API routes.
- Do not push incomplete or dirty data into HubSpot.
- Do not let reports use stale manager names or guessed unit counts.
- Do not invoice without manager approval.
- Do not remove source provenance.

First milestone:
Build a working vertical slice:
1. User enters one property.
2. Backend creates a fact packet.
3. User previews a First Email report.
4. PDF is generated server-side and archived.
5. Report link pushes to HubSpot.
6. TARA queues any billable template task for manager approval.

Output:
- Working app changes
- Database migrations
- Environment variable list
- Setup manual
- Test checklist
- HubSpot sync mapping
- TARA rate-sheet and invoice queue

6. Opinionated Recommendation

Do not keep expanding Jackie. Jackie should be reduced to reporting from verified facts. Daily Hunt, HubSpot Sync, Media Desk, Report Renderer, and TARA should be separate modules. That is the only way this becomes stable enough for the team to trust and revenue-producing enough to justify the build.
